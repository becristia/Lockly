# Lockly 缺陷修复方案

**记录时间:** 2026-05-24 23:12 +08:00  
**范围:** `Lockly` 前端 App、`backend-pass` 后端 API 和管理台  
**原则:** 主密码不上传，云账号密码不参与本地保险库加解密，后端只保存密文和安全元数据。

---

## 修复优先级

1. 先修阻断云同步的数据问题：`SYNC-001`、`SYNC-UX-003`、`AUTH-UX-001`。
2. 再修主密码、同步授权和迁移安全提示：`SEC-UX-001`、`VAULT-UX-001`、`SYNC-UX-002`、`MIGRATION-001`。
3. 再补完整 Android 自动填充：`AUTOFILL-001`。
4. 最后修页面布局和导航刷新：`VAULT-UX-002`、`NAV-001`、`SYNC-UX-004`、`UX-001`。

---

## 1. 主密码强度规则计算错误

**缺陷 ID:** `SEC-UX-001`  
**优先级:** P1  
**影响:** 创建主密码时误判强弱，可能放过弱密码或错误拦截强密码。

**修复方案:**

- 将主密码强度计算集中到一个策略类，例如 `MasterPasswordPolicy`，创建页、修改主密码页、同步主密码确认页只调用同一套规则。
- 强度规则不要只看长度，至少同时考虑长度、大小写、数字、符号、重复字符、常见弱口令、邮箱/用户名片段。
- UI 的强度条、错误文案、提交按钮状态都由同一个 `PasswordPolicyResult` 驱动，避免显示和实际校验不一致。
- 增加边界测试：11/12/16 位、只有数字、只有字母、重复字符、常见弱密码、强密码。

**验收:**

- 弱主密码不能创建保险库。
- 强主密码能创建保险库。
- UI 强度提示和提交结果一致。

## 2. 新增密码强度没有关联输入值

**缺陷 ID:** `VAULT-UX-001`  
**优先级:** P1  
**影响:** 用户新增条目时看不到当前密码真实强度，生成密码回填后强度也可能不刷新。

**修复方案:**

- 新增/编辑条目的密码输入框监听 `TextEditingController`。
- 每次输入、粘贴、生成器返回密码时重新计算条目密码强度。
- 条目密码强度和主密码强度可以共用评分基础，但文案要区分：主密码要求更高，条目密码偏向网站密码建议。
- 生成器保存或回填后，立即刷新强度组件和保存按钮状态。

**验收:**

- 手动输入密码时强度实时变化。
- 从生成器返回密码后强度立即变化。
- 保存后详情页显示的是最新条目。

## 3. 搜索结果下方出现异常空白

**缺陷 ID:** `VAULT-UX-002`  
**优先级:** P2  
**影响:** 搜索页面视觉断层，可能来自底部导航安全区、空状态容器或列表 padding 叠加。

**修复方案:**

- 检查保险库列表页搜索状态下的 `ListView`、`Sliver`、空状态和底部导航 padding。
- 只保留一处底部安全区补偿，避免搜索结果列表和页面容器重复留白。
- 搜索无结果时使用紧凑空状态；有结果时列表高度由内容自然决定。

**验收:**

- 搜索有结果、无结果、键盘展开、键盘收起四种状态都没有异常大空白。

---

## 4. 新增密码进入生成器后的返回和刷新链路混乱

**缺陷 ID:** `NAV-001`  
**优先级:** P1  
**影响:** 从新增密码进入生成器并保存后需要多层返回，最终首页不刷新。

**修复方案:**

- 区分生成器的两种模式：
  - `picker` 模式：从新增/编辑条目进入，只返回生成的密码，不创建条目。
  - `standalone` 模式：从底部生成器页进入，可以直接创建新条目。
- 从新增密码进入生成器时使用 `Navigator.pop(generatedPassword)` 回填到新增页面。
- 如果用户在生成器中选择“保存为条目”，保存完成后清理中间路由，直接回到保险库首页。
- 保险库首页接收保存结果后强制刷新列表，或由 repository/listenable 驱动自动刷新。

**验收:**

- 新增密码页打开生成器，选择密码后回到新增页且字段已填。
- 生成器保存新条目后直接回到保险库首页。
- 首页立即显示新条目，不需要手动刷新或多次返回。

## 5. CSV 导入设计语义不清

**缺陷 ID:** `MIGRATION-001`  
**优先级:** P1  
**影响:** 用户不清楚 CSV 是明文迁移入口，可能误以为 CSV 会安全保存或上传。

**产品定位:**

CSV 导入是“明文迁移工具”，用于从浏览器或其他密码管理器导出的 CSV 本地导入。它可以保留，但必须明确风险。

**修复方案:**

- UI 文案改为“明文 CSV 迁移”或“从明文 CSV 导入”。
- 导入前显示风险提示：CSV 是明文文件，导入完成后应删除原文件。
- App 只在本地读取 CSV，解析后立即加密写入本地保险库。
- 禁止上传原始 CSV，禁止记录 CSV 内容到日志、管理台、同步事件。
- 预览页默认隐藏敏感列，必要时只显示标题/网站/用户名摘要。
- 导入需要本地保险库已解锁；必要时再次要求主密码确认。

**验收:**

- UI 明确提示 CSV 是明文。
- 后端和管理台看不到 CSV 内容。
- 导入后条目只以本地加密形式保存。

## 6. 注册失败没有错误提示

**缺陷 ID:** `AUTH-UX-001`  
**优先级:** P1  
**影响:** 用户不知道是邮箱格式、密码长度、重复注册还是网络问题。

**修复方案:**

- 前端在提交前做本地校验：
  - 邮箱格式。
  - 云账号密码最短 12 位。
  - 密码不能为空。
- 后端继续返回结构化错误码，不返回敏感细节。
- 前端根据错误码映射用户可理解文案：
  - `VALIDATION_ERROR`: 邮箱或密码格式不符合要求。
  - `EMAIL_ALREADY_EXISTS`: 该邮箱已注册。
  - `INVALID_CREDENTIALS`: 邮箱或密码错误。
  - `USER_DISABLED`: 账号已禁用。
  - 网络失败：无法连接云同步服务。
- 表单字段级错误显示在输入框下方，通用错误显示在按钮上方或 Snackbar。

**验收:**

- 注册失败必有可见错误。
- 422 不再表现为无反馈。
- 错误提示不泄露主密码、token 或后端内部异常。

## 7. 同步授权不能用指纹，主密码不能显示/隐藏

**缺陷 ID:** `SYNC-UX-002`  
**优先级:** P1  
**影响:** 同步确认体验差，主密码输入不可检查。

**安全策略:**

- 主密码仍然是最高可信授权方式。
- 生物识别只能作为本地设备解锁方式，不能上传、不能作为云端恢复凭据。
- 如果支持生物识别同步，只能在本地已启用生物识别且本机安全区可取回 DEK 时使用。

**修复方案:**

- 主密码确认弹窗增加显示/隐藏切换按钮。
- 错误主密码时显示明确错误，不跳转、不吞异常。
- 第一阶段保持“同步必须输入主密码”，先修可见性和错误提示。
- 第二阶段可增加“使用生物识别授权本次同步”：
  - 仅在本机已启用生物识别时显示。
  - 文案说明“生物识别只解锁本机保险库，不会上传主密码”。
  - 失败时回退主密码。

**验收:**

- 主密码输入框可以显示/隐藏。
- 错误主密码不会上传数据。
- 生物识别不会改变零知识边界。

## 8. 云同步登录状态不保存

**缺陷 ID:** `SYNC-UX-003`  
**优先级:** P0  
**影响:** App 退出后台再进入后需要重新登录，云同步不可用。

**修复方案:**

- 云同步凭据持久化：
  - access token、refresh token、过期时间、device id 存安全存储。
  - 邮箱和展示状态可以存普通偏好，但 token 不能存明文 SharedPreferences。
- App 启动或恢复前台时读取凭据：
  - access token 未过期：显示已登录。
  - access token 过期但 refresh token 有效：自动刷新。
  - refresh 失败：清理本地云会话，显示需要重新登录。
- 管理台撤销会话、用户禁用、设备撤销后，App 要显示明确失效状态。

**验收:**

- 后台切回不要求重新登录。
- 重启 App 后仍显示云账号状态。
- 管理台撤销会话后旧 token 不再可用。
- 不保存云账号密码。

## 9. 云同步不可用时点击同步没有提示

**缺陷 ID:** `SYNC-UX-004`  
**优先级:** P1  
**影响:** 用户点击同步无反馈，不知道是未配置、未登录、网络失败还是后端不可用。

**修复方案:**

- 同步按钮点击时先做状态判断：
  - 后端地址未配置。
  - 未登录云账号。
  - 设备未注册。
  - 网络连接失败。
  - 后端健康检查失败。
- 不可用时显示明确提示，并给出下一步动作，如“去登录”“检查后端地址”“稍后重试”。
- 设置页云同步状态区域同步显示当前不可用原因。

**验收:**

- 每种不可用状态都有提示。
- 点击同步不会无响应。
- 后端恢复后可以重新同步。

## 10. 云同步上传 422 导致恢复无数据

**缺陷 ID:** `SYNC-001`  
**优先级:** P0  
**影响:** P0-08 上传失败，P0-09 云恢复被阻塞。

**修复方案:**

- 后端不要用“密文字符串中包含 password/key/secret 等片段”来判断是否泄露明文。合法密文是随机编码，可能自然包含这些片段。
- 后端校验应集中在结构和协议字段：
  - 禁止明文字段名。
  - 校验 item/blob id 格式。
  - 校验 AAD schema 和 mac。
  - 校验 ciphertext/base64 格式、大小和 hash。
  - 禁止完整密文明文显示到管理台。
- 前端上传失败时必须展示后端错误，不得跳转后让用户误以为成功。
- 修复后重跑 P0-08：确认 `vault_items > 0`，有附件时 `vault_blobs > 0`。
- 再重跑 P0-09 单设备恢复。

**验收:**

- `/sync/push-vault` 返回 2xx 或明确冲突，不再误报 422。
- 后端数据库有条目和附件密文。
- P0-09 可以恢复条目和附件。

## 11. 同步后跳转到解锁/密码库页且不展示结果

**缺陷 ID:** `UX-001`  
**优先级:** P1  
**影响:** 用户不知道同步是否成功，和 `SYNC-001` 叠加导致误判。

**修复方案:**

- 同步主密码确认只作为当前操作的弹窗，不应改变全局路由。
- 同步完成后留在设置页云同步区域。
- 成功、失败、冲突、未登录、后端不可用都写入同一个状态组件。
- 如果同步过程中触发自动锁定，先完成当前操作，再提示需要重新解锁；不要静默切页。

**验收:**

- 同步后仍在设置页。
- 页面明确显示同步成功/失败/冲突。
- 失败时不会误认为成功。

## 12. Android 自动填充只跳系统设置，未真正填充

**缺陷 ID:** `AUTOFILL-001`  
**优先级:** P1  
**影响:** 用户点击 Android 自动填充后只进入系统设置。即使启用 Lockly Autofill，登录页面也不会出现可填充的账号密码。

**当前状态:**

- `MainActivity.openAutofillSettings()` 调用 Android 系统的自动填充服务选择页面，这是系统要求，App 不能自己静默启用自动填充服务。
- `LocklyAutofillService.onFillRequest()` 当前直接返回 `callback.onSuccess(null)`，没有解析输入框，也没有返回任何 `FillResponse` 或 `Dataset`。
- 设置页文案已经写明“当前阶段不会直接填充已保存条目”，但从产品角度仍需要完整实现。

**修复方案:**

- 保留“打开系统设置”作为启用入口，并在 UI 中明确说明：第一步需要在系统设置里选择 Lockly 作为自动填充服务。
- 实现原生 Android AutofillService：
  - 解析 `FillRequest` 的 `AssistStructure`，识别用户名、邮箱、密码字段。
  - 读取目标 App 包名、Web 域名或页面标识，作为匹配条件。
  - 未解锁前只返回一个需要认证的填充选项，不提前暴露用户名或密码。
- 增加 Autofill 认证/选择流程：
  - 用户点系统自动填充候选项后，打开 Lockly 的认证页面。
  - 本地使用主密码或已启用的生物识别解锁保险库。
  - 根据域名、包名、标题、用户名搜索本地条目。
  - 用户选择一个条目后，把用户名和密码通过 Android Autofill `Dataset` 返回给系统。
- 安全约束：
  - 不上传任何自动填充上下文。
  - 不在日志记录目标页面、用户名、密码。
  - 填充密码只在本次认证结果中短暂存在。
  - 自动保存新密码功能先不做，避免误收集明文；后续单独设计。
- Flutter 侧补充：
  - 设置页显示“未启用 / 已启用 / 已启用但未完成填充服务实现”的准确状态。
  - 增加自动填充使用说明和失败提示。

**验收:**

- 首次点击设置入口会进入系统自动填充设置，这是正确行为。
- 系统设置中选择 Lockly 后，App 设置页显示已启用。
- 在浏览器或测试 App 的登录页点击账号/密码输入框，系统出现 Lockly 填充候选项。
- 选择候选项后先要求本地认证，再展示匹配条目。
- 选择条目后用户名和密码能填入目标 App。
- 未认证前不暴露密码。
