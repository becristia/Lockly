# Lockly 局域网选择性交换设计

**日期:** 2026-05-25  
**范围:** 只修改 `Lockly`。不修改 `backend-pass`。  
**状态:** 待用户确认后再进入实现计划与代码修改。

## 目标

把当前 App 端云同步体验从本分支移除，改成局域网内两台设备之间的选择性交换。发送端明确选择要交换的密码条目，接收端扫码连接并接收数据；接收端输入发送端主密码完成源数据解密校验，然后使用接收端本地保险库密钥重新加密保存。

设计必须满足以下边界：

- 主密码不上传、不同步、不写入二维码、不写入日志、不持久化。
- 后端不参与本分支同步流程；本分支不再依赖云账号、云设备、云同步状态。
- 不做“一次性全量迁移”，不做后台增量同步；每次交换都由发送端选择数据。
- 接收端只导入无冲突条目；有冲突的条目显示错误并忽略，不覆盖、不合并。
- 两端主密码和主密钥可以不同，导入后数据必须落在接收端本地加密体系内。

## 当前上下文

当前 `Lockly` 已有本地保险库、加密备份、单条目备份、附件加密、TOTP、Passkey 元数据、语言切换、设置页、安全中心和云同步服务。云同步相关入口主要集中在：

- `lib/main.dart`: 创建 `SyncStateDao` 和 `SyncService`，读取 `LOCKLY_SYNC_BASE_URL`。
- `lib/app/app_services.dart`: 暴露云登录、注册、设备管理、云上传下载、同步冲突、远程紧急访问等服务。
- `lib/features/settings/settings_page.dart`: 云同步登录、注册、同步、下载和设备管理 UI。
- `lib/features/security_center/security_center_page.dart`: 云同步状态、冲突、云设备和远程紧急访问入口。
- `lib/core/sync/*`: 云同步 API、模型、凭据、payload 校验和服务。

本地备份体系已经具备可复用基础：`BackupService.exportItemBackup()` 能导出单条目及其附件、历史记录，`BackupService.importBackup()` 能验证源主密码，并在目标保险库密钥不同的情况下重加密导入数据。局域网交换应复用这条安全路径，但需要增加多选导出、冲突预检和会话传输。

## 方案比较

### 方案 A: QR 配对 + 局域网 HTTP 会话 + 选择性加密包

发送端启动一个短期本地 HTTP 会话并显示二维码。二维码只包含会话元数据、一次性 token、传输密钥、地址端口和包指纹，不包含保险库数据。接收端扫码后在局域网内拉取加密包，输入发送端主密码验证并解密，再导入无冲突条目。

优点：符合用户的“发送端生成二维码、接收端扫码接收”预期；支持较大数据和附件；不依赖后端；安全边界清晰。  
缺点：需要新增二维码生成和扫码依赖；局域网设备发现、权限和网络异常需要处理。

### 方案 B: 二维码多段承载完整加密数据

发送端把选中条目的加密包切成多个二维码，接收端连续扫码拼包。

优点：不需要局域网 HTTP 服务。  
缺点：二维码容量低，附件和多条目体验很差；扫码失败恢复复杂；用户操作成本高。

### 方案 C: Wi-Fi Direct / 蓝牙近场同步

用平台近场能力完成设备配对和数据传输。

优点：理论上不需要同一 Wi-Fi；系统级配对感更强。  
缺点：Flutter 跨平台插件和 Android/iOS 权限差异大；实现和测试成本高；不适合作为当前分支的主线方案。

**推荐:** 方案 A。它最贴合本分支的产品目标：本地优先、用户主动选择、无后端、可传输多条记录和附件、实现复杂度可控。

## 产品流程

### 发送端

入口放在设置页的“局域网交换”区域，替换原云同步区域。发送端点击“发送数据”后进入选择页：

1. 展示当前保险库条目列表，支持搜索、标签筛选和多选。
2. 默认选择条目的当前字段：标题、网站、用户名、密码、备注、标签、TOTP secret、Passkey 元数据。
3. 附件默认包含在所选条目内，但页面展示预计大小；用户可以关闭“包含附件”。
4. 密码历史默认不包含；用户可以显式打开“包含密码历史”，打开时显示风险提示。
5. 点击“生成二维码”后创建一次性局域网会话。
6. 页面展示二维码、设备名、条目数量、过期倒计时、连接状态、取消按钮。
7. 会话在首次成功传输、用户取消、App 锁定或过期后立即失效。

### 接收端

入口同在“局域网交换”区域。接收端点击“接收数据”后：

1. 打开扫码页，扫描发送端二维码。
2. 校验二维码 schema、版本、过期时间、包指纹字段和会话地址格式。
3. 连接发送端局域网地址并拉取传输包。
4. 输入发送端主密码；输入框支持显示/隐藏切换。
5. App 用发送端主密码验证源备份 envelope 和 manifest。
6. 解密源条目元数据，和本地已解锁保险库做冲突判断。
7. 无冲突条目导入本地并用接收端当前 vault DEK 重新加密。
8. 冲突条目不写入数据库，结果页显示导入数量、忽略数量和冲突列表。

接收端必须已经解锁本地保险库。生物识别只负责本机解锁，不替代发送端主密码。

## 数据范围

第一版交换的数据单位是“密码条目”。每个条目包含：

- `PasswordEntry` 当前字段：title、website、username、password、notes、tags。
- TOTP secret。
- Passkey 准备阶段元数据。
- 与该条目关联的 active attachment blobs，除非发送端关闭“包含附件”。
- 密码历史，仅在发送端显式选择“包含密码历史”时加入。

软删除条目、回收站条目、云同步游标、云设备、云账号、远程紧急访问授权不参与局域网交换。

## 冲突规则

接收端用内容身份判断冲突，不依赖条目 ID，因为两端本来可能来自不同保险库。

冲突键为：

```text
normalize(title) + "|" + normalizeWebsite(website) + "|" + normalize(username)
```

规则：

- `normalize`: trim 后转小写，连续空白折叠为单个空格。
- `normalizeWebsite`: trim、转小写、去掉 `http://` / `https://`、去掉开头 `www.`、去掉末尾 `/`，保留 host 和 path。
- 接收端本地 active 条目存在相同冲突键时，传入条目标记为冲突并忽略。
- 同一传输包内多个传入条目有相同冲突键时，只接受第一个无本地冲突条目，其余标记为包内重复并忽略。
- 空 title、website、username 不跳过冲突判断；空字段按空字符串参与拼接，避免“无网站但同账号”的条目被重复导入。

冲突结果只显示必要元数据：标题、网站、用户名、冲突原因。不会显示密码、TOTP secret、附件内容或 Passkey 私密字段。

## 加密与会话安全

局域网交换采用两层保护。

第一层是保险库备份 envelope：

- 发送端生成选中条目的备份包。
- 条目和附件仍按发送端 vault DEK 加密。
- 源 vault DEK 仍由发送端主密码派生的密钥保护。
- 接收端必须输入发送端主密码，才能验证源 envelope 和 manifest，并解密源条目。
- 接收端导入时不保留发送端 envelope，而是使用本地已解锁 vault DEK 重新加密。

第二层是一次性传输 envelope：

- 发送端为每次会话生成 256-bit 随机 `transfer_key` 和 256-bit 随机 bearer token。
- 备份 JSON 在发送端用 `transfer_key` 做 AES-GCM 传输加密。
- 二维码携带 `transfer_key`、token、session id、地址端口、过期时间、包 SHA-256 指纹和 schema 版本。
- HTTP 请求必须带 token；token 错误、过期、重复下载都拒绝。
- 接收端下载后先校验传输 envelope 和包指纹，再提示发送端主密码。
- QR 不包含主密码、本地 DEK、明文条目或完整备份内容。

这样即使局域网流量被旁路监听，监听者也拿不到可离线破解的备份 ciphertext；必须同时看到二维码并抢在过期前使用 token 才能取得传输包。即便取得传输包，仍需要发送端主密码才能解密保险库数据。

## QR Payload

二维码使用 JSON，外层再做 base64url 或紧凑字符串编码，schema 固定为：

```json
{
  "schema": "lockly-lan-transfer-v1",
  "host": "192.168.1.20",
  "port": 49152,
  "session_id": "uuid",
  "token": "base64url-32-bytes",
  "transfer_key": "base64url-32-bytes",
  "package_sha256": "hex-sha256",
  "selected_count": 3,
  "expires_at": "2026-05-25T12:00:00Z",
  "sender_name": "Pixel 8"
}
```

二维码只服务短期配对。`expires_at` 默认 5 分钟，用户可以刷新二维码重新开会话。

## 传输 API

发送端本地 HTTP server 只暴露最小 API：

- `GET /v1/transfer/{session_id}`  
  Header: `Authorization: Bearer <token>`  
  Response: 传输 envelope JSON，包含 `nonce`、`ciphertext`、`mac`、`content_length`、`package_sha256`。

约束：

- server 只监听当前会话生命周期。
- 默认绑定 `0.0.0.0`，二维码选择一个可达的局域网 IPv4 地址展示；如果多网卡，发送端允许切换地址。
- 单次成功下载后会话失效。
- App 进入锁定态、后台超时、发送端取消或倒计时结束时关闭 server。
- server 不提供列表接口、不提供健康信息、不写出 payload 到磁盘。

## App 结构调整

### 删除云同步 App 入口

本分支删除或迁出以下 App 端能力：

- 云账号注册、登录、退出和登录态持久化。
- 云设备列表、重命名、撤销。
- 云同步上传、云下载、云冲突展示。
- `LOCKLY_SYNC_BASE_URL` 配置入口。
- 设置页和安全中心的云同步可见区域。
- 依赖后端同步服务的远程紧急访问入口。

Android `INTERNET` 权限保留，因为局域网 HTTP 传输仍需要网络权限。Android 系统云备份继续禁用。

### 新增局域网交换模块

建议新增模块：

- `lib/core/lan_sync/lan_transfer_models.dart`: QR payload、传输 envelope、结果模型、冲突模型。
- `lib/core/lan_sync/lan_transfer_crypto.dart`: transfer key AES-GCM、hash、token 校验工具。
- `lib/core/lan_sync/lan_transfer_server.dart`: 发送端短期 HTTP server。
- `lib/core/lan_sync/lan_transfer_client.dart`: 接收端下载和 envelope 校验。
- `lib/core/lan_sync/lan_transfer_service.dart`: 多选导出、源密码验证、冲突预检、重加密导入 orchestration。
- `lib/features/lan_sync/lan_sync_page.dart`: 入口页。
- `lib/features/lan_sync/lan_send_page.dart`: 发送端选择条目与二维码页。
- `lib/features/lan_sync/lan_receive_page.dart`: 扫码、输入发送端主密码、结果页。

### 复用与改造备份层

备份层保留为底层安全 envelope，但需要增加选择性能力：

- 新增多条目导出 API，支持条目 ID 列表。
- 多条目导出支持是否包含附件、是否包含历史。
- 新增 conflict-aware import API：先验证完整源包，再只导入无冲突条目。
- 导入无冲突条目时复用现有“不同 vault envelope 下重加密”的逻辑。
- 保留普通加密备份导入导出；它是本地手工备份能力，不属于云同步。

## 错误处理

用户可见错误按场景明确区分：

- 二维码过期：提示重新生成二维码。
- 网络不可达：提示两台设备需要在同一局域网，允许重试或重新扫码。
- token 错误或会话失效：提示发送端会话已失效。
- 包指纹不匹配：提示传输内容校验失败，不导入任何数据。
- 发送端主密码错误：提示主密码不正确，不导入任何数据。
- 本地保险库锁定：提示先解锁本机保险库。
- 全部冲突：提示未导入任何条目，并展示冲突列表。
- 部分冲突：展示导入数量和忽略数量。

任何失败默认不落库；只有通过源密码验证、manifest 校验、冲突过滤后的无冲突条目才写入。

## 国际化与 UI

新增所有可见文本必须放进中文和英文字符串类，不在 Widget 中硬编码。设置页将云同步区域替换为“局域网交换”，安全中心保留本地安全、备份、自动填充、生物识别、附件和密码健康等本地能力，不再展示云账号或云设备状态。

UI 风格沿用现有 `SecureSection`、`SecurePanel`、`SecureGradientButton`、`SecureVisualBackground`。发送页和接收页必须优先展示当前任务状态，而不是说明型落地页。

## 测试策略

实现阶段需要覆盖：

- QR payload 不包含主密码、DEK、明文密码、TOTP secret、附件明文。
- transfer envelope 使用随机 key 和 nonce；同一数据两次生成 ciphertext 不同。
- token 错误、过期、重复下载都拒绝。
- 发送端主密码错误时不导入任何数据。
- 包指纹不匹配时不导入任何数据。
- 无冲突条目导入后能用接收端本地主密码解锁读取。
- 两端主密码不同、vault envelope 不同时，导入数据被重新加密到接收端 vault。
- 本地已有相同冲突键时传入条目被忽略。
- 包内重复冲突键只导入第一个，其余忽略。
- 设置页不再出现云登录、云注册、云同步、云设备入口。
- 安全中心不再展示云同步状态或远程紧急访问入口。
- `flutter analyze` 无警告以上问题，相关 `flutter test` 通过。

## 手工测试主线

1. 设备 A 创建保险库并新增 3 条记录，其中 1 条带 TOTP，1 条带附件。
2. 设备 B 创建不同主密码的保险库，并新增 1 条会和设备 A 冲突的记录。
3. 设备 A 选择 3 条记录生成二维码。
4. 设备 B 扫码，输入错误的设备 A 主密码，确认不会导入。
5. 设备 B 输入正确的设备 A 主密码。
6. 设备 B 结果页显示 2 条导入、1 条冲突忽略。
7. 设备 B 锁定并用自己的主密码重新解锁，确认导入的 2 条记录可读取，冲突记录未覆盖。
8. 设备 A 原会话二维码再次扫码，确认会话已失效。
9. 设备 A 关闭附件开关再发送，设备 B 确认记录导入但附件不导入。

## 确认点

本文档固定以下决策，除非用户要求调整：

- 本分支只改 `Lockly`。
- 云同步 App 入口和后端依赖从本分支移除。
- 局域网交换只按用户选择的数据执行，不做全量迁移，不做增量同步。
- 接收端必须输入发送端主密码。
- 接收端用自己的本地 vault key 重新加密保存。
- 冲突条目忽略，不覆盖、不合并。
- 附件默认随所选条目传输，密码历史默认不传。
